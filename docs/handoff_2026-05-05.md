# Handoff v4 — NRP, Sensitivity, Direction Choices

Updated 2026-05-05.

## What this session added

**Honest answer to the NRP question:** compute is *not* the current bottleneck for Gaussian-plume calibration. The 200-sample sensitivity analysis ran in 9 seconds locally; v2's NNLS ran in milliseconds. NRP only matters if we're doing something different — Bayesian posteriors with HYSPLIT in the loop, leave-one-event-out CV, Sobol sensitivity at scale, synthetic event generation for alert training. Five workloads laid out in `nrp_calibration_plan.md`.

**Concrete demonstration:** ran a real sensitivity analysis on the emissions model (200 LHS samples × 11 parameters × 38 sources × 3 receptors × 72 hours = ~1.7M source-hour-receptor evaluations). Two findings worth your attention:

1. **`f_arch_estuary` is the single most influential parameter**, with Pearson r = -0.64 against NESTOR's timing correlation. That's a strong first-order signal that the estuary sources are too heavy in the v2 fit.
2. **IB CIVIC CTR's timing prefers drain-dominated emissions** (r = +0.30 for `f_arch_drain`, r = -0.35 for `f_arch_estuary`). This *contradicts* v2's NNLS attribution. v2 fit IB's magnitude well via estuary weight, but at the cost of phase. The diurnal modifier we're adding in v3 is the right tool — `diel_amplitude` also shows -0.33 correlation with NESTOR's timing.

These are findings that the existing v2 calibration loop literally cannot produce because point estimates don't tell you which parameters drive the residuals. Sensitivity analysis does.

## Where compute genuinely helps

In rough order of value-per-CPU-hour:

1. **Bayesian MCMC** for parameter posteriors — gives you uncertainty intervals on every emission rate. ~6,500 CPU-hours with HYSPLIT.
2. **Leave-one-event-out CV** — gives you defensible predictive performance numbers. ~3,000 CPU-hours.
3. **Sobol sensitivity at full scale** — gives you proper variance decomposition. ~10 CPU-hours with Gaussian plume; 1,800 with HYSPLIT.
4. **Synthetic event corpus** for alert system — gives you robustness on rare regimes. ~150 CPU-hours.
5. **Daily EnKF updates** — operational state-space model. ongoing small load.

Architecture proposal: Railway for serving (always-on, low-latency), NRP for batch (parallel, persistent results), object store for shared state. Both clusters use the same package code; only the dispatch differs.

## Direct guidance for next steps

If I had to pick one workload to deploy on NRP first, it would be **Sobol sensitivity at full scale (workload #3)** with the Gaussian plume backend. Rationale:

- It's the lowest-risk plumbing test for the K8s/Argo path. Container, job spec, results retrieval — all small.
- It produces a science deliverable (proper Sobol indices) that the local 200-sample LHS approximates but doesn't deliver.
- It costs ~10 CPU-hours total. If the K8s pipeline breaks, you've lost minutes, not days.
- Once the pipeline works, swapping to HYSPLIT and running #1 (MCMC) or #2 (CV) is a backend change, not an architecture change.

The wrong first move would be deploying HYSPLIT-in-the-loop MCMC. Too many unfamiliar moving parts at once. Get the pipeline working with cheap workloads, then upgrade backend.
